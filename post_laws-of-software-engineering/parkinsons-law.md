# Parkinson's Law

**Definition:** Work tends to stretch out to occupy whatever time has been set aside for it.

**Category:** Planning

## Takeaways
- Long timelines invite procrastination, scope creep, and over-polishing rather than faster delivery.
- With loose deadlines, teams often delay starting or spend the surplus time re-debating minor details.
- Tight-but-realistic deadlines create focus and pace without tipping into burnout.
- Agile time-boxing (sprints, fixed iterations) directly structures work to resist this drift.
- Compressing deadlines too far collides with Hofstadter's Law, so the tension has to be managed, not eliminated.

## Why it matters
Given a generous timeline, people rarely finish early. Instead, extra hours get absorbed by exploring alternatives, gold-plating features, or simply starting later. The task eats the budget, even when it did not really need to.

Setting a clear, moderately tight deadline forces trade-offs earlier and surfaces the essential work faster. This is one reason fixed-length sprints and iteration cycles hold up so well: the box itself does the pacing. The caveat is that shrinking the box too aggressively runs straight into estimation error, so the goal is a deadline that creates urgency without becoming fiction.

## Examples
- **Module build-out:** A developer given one or two months to build a component often experiments and polishes the full window, whereas the same work scoped to a week frequently ships in a week.
- **Meetings:** An all-day meeting tends to use the whole day, while the same agenda in a two-hour block is usually done on time because attendees stay focused.

## When to apply
Use this law during sprint planning, roadmap sizing, and deadline setting. If a task's allocated time feels generous "just in case," that is the trigger - tighten it, carve it into smaller deliverables, or time-box an initial spike. It is also useful when scheduling meetings, code reviews, and design sessions: set an end time and hold it.

Balance it against Hofstadter's Law during estimation. The target is a deadline short enough to concentrate effort but long enough that the team isn't cutting corners on real complexity.
