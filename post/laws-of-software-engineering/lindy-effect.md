# The Lindy Effect

**Definition:** For non-perishable things like technologies and ideas, the longer they've already lasted, the longer they're likely to keep lasting.

**Category:** Decisions

## Takeaways
- Time filters out weak ideas; survival is evidence of robustness.
- Favor fundamentals (algorithms, core languages, protocols) over frameworks with unclear staying power.
- "Boring" technology is often a feature, not a limitation — it's proven.
- Most "new" ideas are reworkings of older ones; recognizing that saves learning cost.
- Use established tools as a hedge against churn and obsolescence.

## Why it matters
Software culture rewards novelty, but novelty is risky: most new frameworks don't make it to year five. The Lindy Effect inverts the usual instinct — instead of asking "is this the newest?" ask "has this already survived a decade of production use?"

The skills and technologies with the longest track records tend to keep paying off. Data structures, SQL, Unix tooling, TCP/IP, and core language features evolve slowly because they've already been stress-tested by millions of users over decades. Betting on them is a low-drama way to avoid the obsolescence treadmill.

## Examples
- **COBOL:** Born in 1959, still running a large share of global banking and financial transactions. Outlived countless "modern replacements."
- **SQL vs. JS frameworks:** SQL (1970s) is still the default way to query relational data. Meanwhile AngularJS, Ember, Knockout, and others rose and faded in the time SQL barely changed.

## When to apply
Use this during technology selection, hiring prioritization, personal learning plans, and architectural decisions about long-lived systems. Whenever stability and maintainability matter more than being cutting-edge — core infrastructure, compliance-sensitive systems, long-horizon platforms — bias toward what has already lasted.

Pair with hype-cycle thinking: the Lindy Effect is a good counterweight when the team is tempted by something at peak buzz.
